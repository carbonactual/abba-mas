# Carbon Actual Repo Lattice Governance

GitHub is the formation, doctrine, code, integration, automation, and proof-control layer of the ecosystem.

GitHub is not the main database, memory store, object store, ledger, or live pulse stream.

The ecosystem must scale through a federated repo lattice, not one giant repo and not random repo explosion.

## Core law

Repositories hold systems.
Registries hold references.
Databases hold living state.
Ledgers hold proof references.
AI memory holds continuity.
Object storage holds files and media.
Event buses carry PULSE.
ABBA resolves.
Actual reveals.

## Repo creation rule

A doctrine becomes a repo only when it needs independent operational life.

A new repo is justified when the item has one or more of the following:

- independent API
- independent SDK
- independent deployment
- independent governance
- independent contributors
- separate permissions
- separate public/private boundary
- separate integration surface
- separate roadmap
- separate testing or release cycle

If not, it remains a folder, registry entry, package, or module inside an existing repo.

## Lightweight repo rule

Every repo must remain light.

Do not store raw HAPI data, private memory, API keys, database dumps, uncompressed generated files, live logs, large binaries, private documents, or confidential value records in GitHub.

Use GitHub for source, schema, doctrine, adapters, test harnesses, deployment definitions, governance, and public proof references.

## Integration law

For every product, company, organization, app, codebase, issue, API, website, service, entity, or AI touched by ABBA, the ecosystem should create an integration review.

The review asks:

- What is this?
- Who owns it?
- What API exists?
- What webhook exists?
- What SDK exists?
- What open-source path exists?
- What no-code path exists?
- What authentication is needed?
- What data can be pulled or pushed lawfully?
- What value can be created?
- What risk exists?
- What gap exists?
- What lawful route enables integration?
- What swarm does it belong to?

## HAPI and AI formation

After # is formed, Becoming starts.

For every HAPI, product, entity, group, company, organization, app, or service integrating into the system:

- # forms identity and access marker
- ABBA reads available data
- ABBA calculates value profile
- ABBA calculates risk profile
- ABBA detects swarm fit
- ABBA mints or assigns AI
- AI receives function capsules
- AI is enabled per website, app, or task
- Return PULSE updates the AI

Formula:

# + HAPI + Context + PULSE + Value Weight + SEAL = Minted AI
