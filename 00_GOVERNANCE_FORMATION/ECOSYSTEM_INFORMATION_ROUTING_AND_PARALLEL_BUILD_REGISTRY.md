# Ecosystem Information Routing and Parallel Build Registry

This registry prevents the ecosystem from becoming trapped in one doctrine, one product, or one repository.

Every new instruction, idea, law, feature, integration, product decision, interface concept, proof, dataset, workflow, or operating rule must be classified and routed to its correct destination while preserving links to the full ecosystem.

## Routing principle

ABBA must process every new input through five questions:

1. What kind of information is this?
2. Which ecosystem layer owns it?
3. Which product or repository must implement it?
4. Which other layers depend on it?
5. Does it require doctrine, schema, code, integration, test, deployment, or monitoring?

## Parallel build rule

Doctrine and build may proceed together.

A product does not wait for every doctrine to be complete.
A doctrine discovered during implementation must be routed back to abba-mas.
A reusable implementation discovered inside a product must be promoted to shared infrastructure.

The cycle is:

idea
→ classify
→ route
→ build
→ test
→ observe
→ return PULSE
→ update doctrine or implementation
→ continue

## Cross-link rule

Every routed item must preserve:

- source instruction
- owner layer
- destination repository
- dependent layers
- implementation status
- proof link
- return PULSE

## Build states

Each item must be marked as one of:

- captured
- classified
- routed
- designed
- building
- tested
- deployed
- live
- monitored
- blocked
- superseded

## No-stagnation law

The ecosystem must not remain indefinitely in explanation mode.

When an idea is sufficiently clear to test, it moves into implementation.
When implementation reveals missing doctrine, that doctrine is added without stopping the whole build.
