# ABBA MAS Control and Formation Rules

ABBA MAS is the living integration and swarm command center of Carbon Actual.

It is always growing, but the workspace must remain minimal, decompressed, and controlled.

## Core position

ABBA MAS controls formation, integration, swarm discovery, open-source paths, white-label paths, AI agents, APIs, SDKs, low-hassle tools, free tools, provider registries, and ecosystem routing.

ABBA MAS should grow through modules first, not through uncontrolled repo creation.

## Minimal workspace law

Do not create many branches that cannot be maintained.
Do not create many repos before operational separation is needed.
Do not split doctrines too early.
Do not duplicate what can be referenced.
Do not store heavy or living data in GitHub.
Do not allow workspace sprawl to outrun understanding.

The workspace must remain small enough to reason about, test, secure, and deploy.

## Decompression law

When an idea is large, decompress it into:

1. doctrine
2. schema
3. registry
4. mock route
5. test
6. dashboard reference
7. later extraction decision

Do not jump directly from doctrine to a new repo unless it needs independent deployment, contributors, permissions, release cycle, governance, or security boundary.

## Growth rule

ABBA MAS grows by adding controlled modules.

A module becomes a separate repo only when one of these is true:

- independent deployment is required
- separate permission boundary is required
- separate team ownership is required
- release cycle becomes independent
- security boundary becomes different
- public/private boundary becomes different
- dependency tree becomes too heavy
- the module begins choking ABBA MAS

## Branch rule

Default branch remains the stable trunk.

Use direct small commits for doctrine, schemas, registries, and mock-first routes.

Create branches only for risky code, large rewrites, conflicting changes, external contributors, or production changes that require review.

Delete or merge temporary branches quickly.

No branch should exist without a clear purpose, owner, and expected closure.

## Corrected formation sequence

Signal enters.
ABBA reads.
PULSE is formed.
PULSE is minted.
PULSE is tokenized.
Token is valued.
Value is read.
Field is checked.
Action route is created.
PULSE is released.
PULSE is taken.
PULSE is shot.
PULSE is moved.
PULSE has gone.
Value is revealed.

## Meaning of release

The action route is the release point.

Before the action route, PULSE is still inside formation.
After the action route, PULSE has gone into movement.
Once PULSE has gone, value can be revealed according to SEAL, proof, field state, and Actual readiness.

## Integration sequence

For every product, company, app, tool, repo, issue, API, SDK, model, agent, open-source project, or white-label path:

- identify owner
- identify function
- identify API or integration route
- identify lawful access path
- identify cost
- identify risk
- identify value
- identify field
- classify token
- create registry entry
- test with mock route first
- escalate only after proof

## Final law

ABBA MAS is allowed to grow endlessly, but the workspace must not become chaotic.

Grow by compression.
Expand by modules.
Split only by necessity.
Control before scale.
Formation before activation.
PULSE before value.
Pulse minting before pulse release.
Token valuation before value reveal.
Field check before action route.
Action route releases PULSE.
Value reveals after PULSE has gone.
